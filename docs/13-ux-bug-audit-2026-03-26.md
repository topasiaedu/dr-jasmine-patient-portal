## Bug Inventory

### üî¥ Critical (P0)

#### BUG-01 ‚Äî NumberStepper shows blank display on initial load
- **Affects:** `/p/demo/log` ‚Äî all 7 reading steps
- **Symptom:** The number between `‚àí` and `+` is invisible when the step first loads.
  The `‚àí` button is disabled. Users do not know what value they are setting.
  Tapping `+` causes a jump to `defaultValue` (e.g., 5.5) rather than incrementing from 0.
- **Root cause:** `components/patient/NumberStepper.tsx` line 44‚Äì45:
  ```ts
  const [inputValue, setInputValue] = useState(
    value === 0 ? "" : value.toString()
  );
  ```
  When `value` is `0` the display renders an empty string. Because `min = 0` the
  `‚àí` button is also disabled (`value <= min`). The `defaultValue` prop exists
  but is never shown until `+` is tapped.
- **Fix:** Initialise `inputValue` with `defaultValue` (when provided) instead of `""`.
  Change the disabled condition on `‚àí` to also account for the `value === 0 &&
  defaultValue` case so the button is usable from the start.

#### BUG-02 ‚Äî Log success page redirects before patient can read it
- **Affects:** `/p/demo/log/success`
- **Symptom:** Navigating directly to `/p/demo/log/success` (or arriving from the
  log flow) results in an immediate redirect to home. The "Done!" message and
  confetti animation are never seen.
- **Root cause:** `app/p/demo/log/success/page.tsx` ‚Äî the 3-second `setInterval`
  starts the instant the component mounts, with no initial render delay. On fast
  devices (and in testing) the page is loaded, the interval fires, and the push to
  home happens before the screenshot or the patient's eye can register it.
- **Fix:** Add a 500 ms `setTimeout` before starting the countdown interval.
  Change `useState(3)` to `useState(4)` so the visible countdown shows "4‚Ä¶3‚Ä¶2‚Ä¶1"
  giving a full 4 real seconds of display time before the redirect.

### üü° Significant (P1)

#### BUG-03 ‚Äî Validation error persists after required fields are filled
- **Affects:** `/p/demo/onboarding` Step 1 (Personal Details)
- **Symptom:** Clicking Next on an empty form shows "Please fill out your name,
  age, and sex." After all three fields are correctly filled, the red error message
  remains visible below the form until the button is clicked again successfully.
  There is no live clearing of the error.
- **Fix:** Clear the error string any time all required fields become valid
  (inline `useEffect` or `onChange` handlers), not only on the Next button click.

#### BUG-04 ‚Äî Admin Schedule ‚Äî blank button on "Scheduled" appointment cards
- **Affects:** `/admin/schedule`
- **Symptom:** For appointments with status "Scheduled", a second button element
  appears next to "View Profile" but renders with no visible label or content.
- **Fix:** Identify and fix the conditional rendering for the "Start Consultation"
  or "Join Zoom" button that is generating an empty element.

#### BUG-05 ‚Äî Appointment end time shows "TBD"
- **Affects:** `/p/demo/pending` (confirmation card) and `/p/demo/appointment`
- **Symptom:** The time field reads "10:00 AM ‚Äì TBD". This looks unfinished and
  untrustworthy in a medical context.
- **Fix:** Derive the end time from a default session duration of 30 minutes
  (`endTime = startTime + 30 min`), or display only the start time.

#### BUG-06 ‚Äî Greeting ARIA text has extra space: "Good morning , Lily üëã"
- **Affects:** `/p/demo/home`
- **Symptom:** The `h1` element's accessible name contains a spurious space before
  the comma. Visual rendering may look correct but assistive technologies read it
  incorrectly.
- **Fix:** Remove the extra space in the greeting string interpolation.

### üüÝ UX Issues (P2)

#### UX-01 ‚Äî Home page task card does not update after readings are logged
- The "Have you logged your readings today?" card always shows ‚Äî even after the
  patient completes the log flow. There is no "Done ‚úì" state on the home page.
  `localStorage` key `demo_reading_today` is set after logging but the home page
  does not read it to update the card.

#### UX-02 ‚Äî "Book a new appointment" button shown when appointment already exists
- `/p/demo/appointment` renders a teal "Book a new appointment" CTA even when an
  active appointment is booked. This confuses patients into thinking they must
  re-book.

#### UX-03 ‚Äî Photo logging option is non-functional with no feedback
- On `/p/demo/log`, tapping "Take a photo" leads nowhere. There is no "coming
  soon" message or disabled state. Patients who select it are left confused.

#### UX-04 ‚Äî Zoom join note text is below the 14 px minimum
- The note "(Available 15 min before appointment)" and similar hint strings appear
  to render at ‚â§ 12 px. The design spec mandates 14 px minimum for all patient-
  facing text (nav labels excepted).

#### UX-05 ‚Äî Disabled "Join on Zoom" button has inconsistent styling across pages
- Pending page: full-width muted-gold button.
- Home page: teal outline, greyed.
- Appointment page: full-width muted-gold button.
  All three should use the same disabled visual treatment.

#### UX-06 ‚Äî No Sonner toasts on admin saves
- "Save Note" (consultation page) and "Save & Publish" (guide builder) complete
  silently. The Sonner toaster is installed; it is just not called.

#### UX-07 ‚Äî Step progress dots have no text labels or aria descriptions
- In both onboarding and log flows, progress is shown as plain dots. Screen-reader
  users hear nothing. Sighted patients cannot tell how many steps remain or what
  each step is named.

#### UX-08 ‚Äî No "Skip to main content" link on any page
- Required by the design spec (doc 08-ui-ux.md). Currently absent from every page.

#### UX-09 ‚Äî Password field has no show/hide toggle
- Admin login password input has type="password" with no reveal button. Elderly or
  low-tech users frequently mistype passwords when they cannot see what they typed.

#### UX-10 ‚Äî Hero greeting font is Plus Jakarta Sans, not DM Serif Display
- Design spec: the patient greeting name ("Good morning, Lily") should use the
  brand serif (`--font-display`). Currently the entire heading is Plus Jakarta Sans.

#### UX-11 ‚Äî Sex selection uses full-width stacked buttons, not a toggle group
- Three full-width buttons (Male / Female / Prefer not to say) stacked vertically
  do not visually communicate "pick one". The selected state (light teal bg) is
  subtle. A horizontal pill toggle group with a clear selected state would be clearer.

#### UX-12 ‚Äî Admin dashboard stat cards lack visual priority
- All four cards are identical in size, weight, and colour. "Readings to Review (7)"
  is more urgent than "Guide Adherence (84%)" but looks identical.

#### UX-13 ‚Äî Admin schedule card layout breaks on long patient names
- "Lily Tan" + "Scheduled" badge wraps to two lines on the schedule card, making
  the card taller than intended and the layout uneven.
